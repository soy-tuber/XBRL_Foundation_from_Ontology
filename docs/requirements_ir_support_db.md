# 要件書: 飲食業 IR/法務支援 DB システム (仮称)

## 1. 目的と背景

上場準備中の飲食業企業の IR 部門・CFO・法務が、有価証券報告書等の開示書類作成と適時開示義務の判断を効率化するための社内ツール。将来的に他業種への展開、個人投資家向けマスク公開を視野に入れる。

## 2. スコープ

### 2.1 含むもの

- 飲食業上場企業の有価証券報告書 (およびその訂正報告書) を対象とした構造化 DB
- 以下4機能を備えた検索・支援 UI
  - 有報作成支援
  - 決算説明資料作成支援 (第2フェーズ、Google Drive ベース)
  - コンプラ/リスクチェッカー
  - 適時開示ヒットチェッカー/アラート

### 2.2 含まないもの (明示的除外)

- 決算短信の DB 化 (有報の蒸留であるため不要)
- 株価データ (J-Quants 再配布制約の回避、および今回の用途に不要)
- リアルタイム株価・板情報
- 決算説明資料からのデザイン自動生成 (将来フェーズ、本要件では保留)

## 3. フェーズ構成

### フェーズ1: 有報ベースの基盤 DB + 4機能プロト

- 本要件書の主対象
- デリバラブル: Streamlit プロト + SQLite DB

### フェーズ2: 決算説明資料検索

- Google Drive に決算説明資料を格納し検索可能化
- 実装方法は既存ノウハウで確立済み、本要件書では詳細不要

### フェーズ3 (保留): 数式入りスプレッドシート/GAS コード生成

- 決算説明資料の大量データから推論
- アウトプットは Sheets + GAS コード (デザインは会社依存のため深追いしない)
- フェーズ1,2の稼働実績を見て着手判断

## 4. データ要件

### 4.1 データソース

- EDINET API v2 (金融庁公式、無料、API キー認証)
- 対象書類種別:
  - 有価証券報告書 (docTypeCode=120)
  - 訂正有価証券報告書 (docTypeCode=130)
  - 臨時報告書 (docTypeCode=160) — 適時開示アラートの参考事例用
  - 大量保有報告書 (docTypeCode=350) — コンプラ参照用

### 4.2 対象企業

- 飲食業上場企業 約60社 (具体リストは実装時に絞り込み)
- 選定は各種業種分類サイト・EDINET業種コードを参照

### 4.3 対象期間

- EDINET API 上限 10年分
- フェーズ1 プロトは直近5年で開始、動作確認後に10年へ拡張可能

### 4.4 対象セクション

有報の主要セクション全てを取得:

- 事業の内容
- 事業等のリスク
- MD&A (経営者による財政状態、経営成績等の分析)
- 経営方針、経営環境及び対処すべき課題
- サステナビリティに関する考え方及び取組
- 従業員の状況 (人的資本含む)
- コーポレート・ガバナンスの状況
- 役員の状況
- 役員の報酬等
- 大株主の状況
- 株式の保有状況 (政策保有株式)
- 関連当事者情報
- セグメント情報
- 財務諸表本体 (BS/PL/CF、連結・単体)

### 4.5 データ品質要件

- 純度: XBRL タグ残骸・HTMLエンティティ・目次/ページ番号・脚注ゴミの除去
- 可用性: チャンク本体はクリーンなテキスト、metadata (提出日、期間、訂正フラグ、タクソノミ世代等) は濃厚に保持
- タクソノミ世代対応: 2014/2019/2023 の要素名揺れを正規化辞書でマッピング
- 訂正報告書の扱い: `is_latest` フラグ + `superseded_by` リンクで最新版を特定可能に

## 5. スキーマ設計

```
companies
  - edinet_code (PK)
  - sec_code
  - company_name
  - industry_code
  - fiscal_year_end

documents
  - doc_id (PK)
  - edinet_code (FK)
  - doc_type_code
  - period_start, period_end
  - submit_date
  - is_amended
  - superseded_by
  - is_latest
  - taxonomy_version

sections
  - section_id (PK)
  - doc_id (FK)
  - section_code (正規化済み、世代またいで同一)

  - section_order

sections_fts (FTS5 仮想テーブル)

financial_figures
  - fig_id (PK)
  - doc_id (FK)

```

## 6. システム構成

### 6.1 プロト開発環境 (Soy さん手元)

- 開発: Claude Code + Gemini CLI (ローカル LLM 的に対話利用) + ローカル Nemotron (必要時)
- 開発機: RTX 5090 機 + Chromebook
- 出力: Streamlit プロト + SQLite DB + セットアップスクリプト一式

### 6.2 本番環境 (在籍企業)

- 会社サーバー用 PC にインストール (社内LAN運用)
- LLM: Gemini API (Google Workspace 契約)
- セキュリティは会社の既存ポリシーに準拠

## 7. 4機能の仕様概要

### 7.1 有報作成支援

- ドラフト作成中のセクションに対応する同業他社の最新記載を横並び表示
- 類似表現サジェスト、前年度自社との差分表示

### 7.2 決算説明資料作成支援 (フェーズ2)

- Google Drive 上の決算説明資料を全文検索
- 既存ノウハウで実装

### 7.3 コンプラ/リスクチェッカー

- 対象セクション: 事業等のリスク、コーポレート・ガバナンス、内部統制
- ルールは context として LLM に投入:
  - 適時開示規則
  - 金商法166条 (インサイダー)
  - 企業内容等開示府令19条 (臨報事由)
  - その他関連法令

### 7.4 適時開示ヒットチェッカー/アラート

- 社内イベント入力 → ルール判定 → 該当する開示義務を提示
- 過去の臨報事例を参照情報として表示
- 運用ポリシー: 過剰検知寄り (見逃しゼロ優先、人間が最終判断)

## 8. 非機能要件

- 社内LAN完結で動作
- Gemini API 以外への外部通信は EDINET API クロール時のみ
- 社外秘情報は Gemini Enterprise の ZDR 契約範囲で処理
- バックアップ: 3-2-1 ルール (既存ノウハウに準拠)

## 9. 成果物

1. SQLite DB (飲食業60社×5年分、フェーズ1完了時点)
2. Streamlit プロトアプリ (4機能 UI)
3. EDINET クロール・XBRL パース・DB 投入の一式スクリプト
4. タクソノミ世代マッピング辞書
5. セットアップ手順書 (会社PC デプロイ用)

- 飲食業特化部分および社内特注ロジックは在籍企業帰属
- 境界は実装時に契約書で明示
